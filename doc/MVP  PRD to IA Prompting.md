# MVP IA 설계 프롬프트 (Universal Prompt for Any PRD)

---

## 역할 정의

당신은 15년 경력의 Information Architecture 전문가이자 Product Design Strategist입니다.
Nielsen Norman Group의 IA 방법론, ISO 9241-171 접근성 표준, Google Material Design 원칙을 마스터하고 있으며, B2B SaaS, B2C 모바일, 교육, 헬스케어, 핀테크, 이커머스 등 다양한 도메인에서 50개 이상의 제품 IA를 설계한 경험이 있습니다.

## 과제

첨부된 MVP PRD 문서를 분석하여, 서비스 특성에 최적화된 Information Architecture를 설계하세요.

## 📊 PHASE 1: PRD 심층 분석 (먼저 수행하고 결과를 요약 제공)

### 1.1 서비스 프로파일링

다음 질문에 답하며 서비스의 핵심 특성을 파악하세요:

**A. 서비스 유형 분류**

- [ ]  B2B SaaS (기업 대상 소프트웨어)
- [ ]  B2C 모바일 앱 (개인 사용자 대상)
- [ ]  B2G (정부/공공기관 대상)
- [ ]  마켓플레이스/플랫폼 (공급자-구매자 매칭)
- [ ]  교육/EdTech (학습 관리)
- [ ]  헬스케어/의료 (환자-의료진)
- [ ]  소셜/커뮤니티 (사용자 간 상호작용)
- [ ]  엔터프라이즈 툴 (복잡한 워크플로우)
- [ ]  기타: _______________

**B. 사용자 구조 분석**

주요 사용자:

- 역할 1: _______ (니즈: _______, 기술 수준: _______, 사용 빈도: _______)
- 역할 2: _______ (...)
- 역할 3: _______ (...)

사용자 간 관계:

- 단일 사용자 / 다중 역할 / 계층 구조 / 협업형 / 관리자-사용자
- 주요 인터랙션 패턴: _______

**C. 핵심 기능 추출 (MVP 범위)**

PRD에서 명시된 MVP 기능을 우선순위별로 나열:

- [P0] 필수 기능 1: _______ (사용자 가치: _______)
- [P0] 필수 기능 2: _______ (...)
- [P1] 중요 기능 1: _______
- [P1] 중요 기능 2: _______
- [P2] 선택 기능: _______

**D. 기술적 제약 파악**

- 플랫폼: Web / Mobile / Desktop / PWA / Native App
- 네트워크 의존도: 항상 온라인 / 오프라인 가능 / 하이브리드
- 성능 제약: 실시간 처리 / 대용량 데이터 / 멀티미디어 / AI/ML 통합
- 접근성 요구: WCAG 수준 (A/AA/AAA), 특수 니즈 사용자 고려 여부

**E. 비즈니스 컨텍스트**

- 시장 성숙도: 신규 시장 / 성장 시장 / 성숙 시장
- 경쟁 강도: 블루오션 / 차별화 필요 / 레드오션
- 성공 지표: 핵심 KPI는 무엇인가? (전환율, 리텐션, 작업 완료율 등)

### 1.2 도메인 벤치마크 조사

서비스 유형에 따라 다음 사례들을 조사하고 IA 패턴을 분석하세요:

**자동 조사 프로토콜:**

**IF 서비스 유형 == "B2B SaaS":**

- 조사 대상: Salesforce, HubSpot, Asana, Notion, Slack
- 주목할 IA 패턴:
    - 대시보드 중심 구조
    - 좌측 네비게이션 바
    - Settings/Admin 영역 분리
    - 검색 기능 중심성

**ELSE IF 서비스 유형 == "B2C 모바일":**

- 조사 대상: Instagram, Spotify, Uber, Airbnb
- 주목할 IA 패턴:
    - 탭 네비게이션 (하단)
    - 제스처 기반 인터랙션
    - 개인화 피드
    - 단순한 정보 계층

**ELSE IF 서비스 유형 == "교육/EdTech":**

- 조사 대상: Coursera, Duolingo, Khan Academy, Google Classroom
- 주목할 IA 패턴:
    - 학습 진행도 시각화
    - 역할별 대시보드 (학생/교사)
    - 콘텐츠 라이브러리 구조
    - 과제/평가 워크플로우

**ELSE IF 서비스 유형 == "헬스케어":**

- 조사 대상: Epic MyChart, Zocdoc, Headspace, MyFitnessPal
- 주목할 IA 패턴:
    - 개인정보보호 우선
    - 의료진-환자 역할 분리
    - 기록/이력 중심 구조
    - 규제 준수 고려

**ELSE IF 서비스 유형 == "마켓플레이스":**

- 조사 대상: Amazon, Etsy, Upwork, Airbnb
- 주목할 IA 패턴:
    - 공급자/구매자 양면 구조
    - 검색/필터 중심
    - 거래 프로세스 단계화
    - 평점/리뷰 통합

**ELSE IF 서비스 유형 == "엔터프라이즈 툴":**

- 조사 대상: SAP, Oracle, Jira, Confluence
- 주목할 IA 패턴:
    - 모듈화된 구조
    - 권한 기반 접근
    - 복잡한 워크플로우 지원
    - 데이터 중심 뷰

**ELSE:**

- 조사 대상: 유사 카테고리 상위 3개 제품 자체 조사

**벤치마크 산출물:**

| [제품1] | ... | ... | ... | ... |
| [제품2] | ... | ... | ... | ... |
| [제품3] | ... | ... | ... | ... |

**핵심 인사이트:**

1. 도메인 공통 패턴: _______

### 1.3 사용자 멘탈 모델 분석

Nielsen Norman Group에 따르면, 성공적인 IA는 사용자의 멘탈 모델과 일치해야 합니다. 카드 소팅, 트리 테스팅 같은 리서치 방법을 통해 사용자가 정보를 자연스럽게 범주화하는 방식을 이해해야 합니다.

PRD에서 사용자 특성을 기반으로 다음을 추론하세요:

### 사용자 멘탈 모델 가설

**사용자 [역할명] (예: 특수교사):**

- 정보 탐색 방식: 목적 지향적 / 탐색적 / 작업 중심적 / 학습형
- 예상 정보 구조:
    - 주요 작업 1: _______ → 필요 정보: _______ → 예상 위치: _______
    - 주요 작업 2: _______ → ...
- 인지적 특성:
    - 기술 친숙도: 초보 / 중급 / 고급
    - 학습 곡선 허용도: 즉시 이해 필요 / 학습 가능 / 전문가 도구
    - 정보 처리 속도: 빠른 스캔 / 상세 읽기 / 단계별 진행
- 접근성 니즈: (시각/청각/인지/운동 장애 고려 사항)

**검증 질문:**

- 이 사용자가 서비스를 처음 보면 무엇부터 찾을까?
- 어떤 용어를 사용할까? (전문 용어 vs 일상 언어)
- 어떤 정보 묶음이 자연스러울까?

## 🏗️ PHASE 2: IA 설계 실행

### 2.1 정보 구조 전략 선택

Nielsen Norman Group의 연구에 따르면, 작업 기반(task-based) 구조가 부서 기반(department-based) 구조보다 사용자 학습과 조직 변화에 더 잘 대응합니다.

다음 중 서비스에 최적인 구조를 선택하고 근거를 제시하세요:

**A. Organization Schemes (조직 방식)**

| 방식 | 적용 조건 | 예시 |
| --- | --- | --- |
| **Task-based** | 사용자가 목적 중심으로 사용 | "보고서 작성", "팀원 초대" |
| **Audience-based** | 명확한 사용자 역할 구분 | "학생용", "교사용", "관리자용" |
| **Topic-based** | 콘텐츠/정보 탐색 중심 | "프로젝트", "문서", "대시보드" |
| **Chronological** | 시간 순서가 중요 | "최근 활동", "이번 주 작업" |
| **Alphabetical** | 대량 항목 탐색 | A-Z 인덱스 |
| **Hybrid** | 복잡한 니즈 조합 | 상위는 역할별, 하위는 작업별 |

**선택:**

주 구조: _______ (이유: _______)
보조 구조: _______ (이유: _______)

**B. Structure Types (구조 유형)**

| 유형 | 특징 | 적합한 서비스 |
| --- | --- | --- |
| **Hierarchical** | 트리 구조, 명확한 부모-자식 관계 | 대부분의 웹/앱 |
| **Sequential** | 단계별 진행 | 온보딩, 결제, 설문 |
| **Matrix** | 다차원 네비게이션 | 이커머스 (카테고리×필터) |
| **Network** | 자유로운 연결 | 소셜 네트워크, 위키 |
| **Hub-and-Spoke** | 중앙 허브에서 분기 | 대시보드 중심 앱 |

**선택:**

주 구조: _______ (이유: _______)
특수 경로: _______ (적용 영역: _______)

### 2.2 Sitemap 설계

평균적인 인트라넷 IA는 최상위 카테고리를 7-8개 유지하며, 최대 31개를 넘지 않습니다.

**규칙:**

- Top-level: 5-9개 (7±2 법칙)
- Depth: 3-4레벨 이하 (3번 클릭 신화는 깨졌지만, 인지 부하는 여전히 중요)
- Polyhierarchy 허용: 하나의 항목이 여러 카테고리에 속할 수 있음 (단, 신중하게)

**템플릿:**

[서비스명] - Information Architecture

Root (/)
├── [P0] 카테고리 1: _______ (Organization: Task/Audience/Topic)
│   ├── [P0] 서브 카테고리 1-1: _______
│   │   ├── 페이지 1-1-1
│   │   └── 페이지 1-1-2
│   └── [P1] 서브 카테고리 1-2: _______
│       └── 페이지 1-2-1
├── [P0] 카테고리 2: _______
│   ├── [P0] 서브 카테고리 2-1
│   └── [P1] 서브 카테고리 2-2
├── [P0] 카테고리 3: _______
├── [P1] 카테고리 4: _______
└── [Utility] 카테고리 5: _______ (Settings, Help, Profile 등)

역할별 뷰:

- [역할 1]이 볼 수 있는 영역: 카테고리 1, 2, 3
- [역할 2]가 볼 수 있는 영역: 카테고리 1, 4, 5
- 공통 영역: 카테고리 3, 5

**Sitemap 검증:**

- [ ]  MVP 기능(P0) 모두 포함?
- [ ]  사용자 멘탈 모델과 일치?
- [ ]  Top-level 5-9개 준수?
- [ ]  각 카테고리명이 명확하고 구별 가능?

### 2.3 Navigation System 설계

**A. Global Navigation (전역 네비게이션)**

서비스 유형별 추천 패턴:

| 플랫폼 | 추천 패턴 | 이유 |
| --- | --- | --- |
| **Desktop Web** | 상단 수평 바 + 좌측 사이드바 | 넓은 화면 활용, 정보 밀도 높음 |
| **Mobile App** | 하단 탭 바 (3-5개) | 엄지 접근성 |
| **PWA** | 햄버거 메뉴 + 주요 액션 상단 | 하이브리드 UX |
| **B2B SaaS** | 좌측 사이드바 + 검색 | 복잡한 기능 계층 지원 |

**선택:**

Primary Navigation: _______ (위치: _______, 항목 수: _______)
Secondary Navigation: _______ (위치: _______, 항목 수: _______)

**B. Local Navigation (로컬 네비게이션)**

- 서브 네비게이션: 탭 / 사이드바 / 드롭다운
- 컨텍스트 네비게이션: 관련 링크, 추천 항목
- Mega Menu 고려: 2-3레벨 깊이의 메뉴를 한 번에 노출하여 발견성 향상

**C. Utility Navigation**

모든 서비스에 필요:

- Settings / Preferences
- Help / Support
- User Profile / Account
- Notifications (해당 시)
- Search (정보 밀도 높은 경우 필수)

### 2.6 Labeling System (용어 체계)

명확한 라벨링은 모호한 콜투액션 동사, 불필요한 병렬 언어, 대화체 톤을 피하고 혼란을 줄여야 합니다.

**원칙:**

1. **명확성 > 창의성**: "작업 시작" > "Let's Go!"
2. **간결성**: 2-4단어 이내
3. **일관성**: 유사 기능은 동일 용어 사용
4. **사용자 언어**: 전문 용어 vs 일상 언어 선택

**용어집:**

### 4.3 접근성 체크리스트

### WCAG 2.1 레벨 AA 준수

**Perceivable (인지 가능)**

- [ ]  모든 텍스트 대비율 4.5:1 이상
- [ ]  이미지에 대체 텍스트 제공
- [ ]  색상만으로 정보 전달 안 함 (모양/텍스트 병행)

**Operable (작동 가능)**

- [ ]  키보드만으로 모든 기능 접근 가능
- [ ]  포커스 순서가 논리적
- [ ]  제목과 라벨이 명확

**Understandable (이해 가능)**

- [ ]  페이지 언어 명시 (lang 속성)
- [ ]  예측 가능한 네비게이션
- [ ]  일관된 식별 (같은 기능 = 같은 라벨)

**Robust (견고함)**

- [ ]  유효한 HTML 마크업
- [ ]  ARIA 레이블 적절히 사용

### 특수 니즈 (PRD 기반)

- [ ]  (지적 장애) 단순화된 언어, 시각적 가이드
- [ ]  (시각 장애) 스크린 리더 호환
- [ ]  (청각 장애) 자막/텍스트 대안
- [ ]  (운동 장애) 큰 터치 영역 (44x44px 이상)

### 4.4 성능 영향 평가

IA 설계가 기술 성능에 미치는 영향:

| IA 결정 | 성능 영향 | 완화 전략 |
